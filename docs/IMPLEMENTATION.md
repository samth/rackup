# Implementation Notes

This document describes the key implementation choices in rackup. It is aimed at contributors and anyone interested in the internals.

## Documentation Ownership

`docs/IMPLEMENTATION.md` is the canonical source of truth for contributor-facing architecture and implementation behavior.

- Any change to architectural behavior must be documented here in the same change.
- `README.md` should stay user-facing (install and usage), and should link here instead of duplicating internals.
- `docs/IMPL_NOTES.md` and files in `docs/plans/` may contain historical context, but should link back to this document for canonical behavior.

## Two distribution modes

Rackup ships in two forms: a prebuilt executable and a source distribution. The choice affects startup and runtime but the core logic is identical.

**Prebuilt executable mode.** `raco exe` + `raco distribute` produce a `rackup-core` binary that embeds the Racket runtime. The shell wrapper (`bin/rackup`) detects `rackup-core` and execs it directly. No hidden runtime directory is needed.

**Source mode.** The source distribution contains the Racket source files under `libexec/`. The shell wrapper prefers the hidden internal Racket installation (`~/.rackup/runtime/`) when present, invoking `racket -U -A <addon-dir> libexec/rackup-core.rkt`. If the hidden runtime is absent, the wrapper can fall back to a system `racket` found on PATH, but only when `RACKUP_ALLOW_SYSTEM_RACKET=1` is set in the environment. This env var gate exists to prevent accidental use of unrelated system Racket installations; it is intended for development and testing. When system racket fallback is active, `-y` is added (to recompile stale bytecode) and a separate addon directory is used. If neither a hidden runtime nor a permitted system `racket` is available, the wrapper fails immediately with an error message suggesting reinstallation.

`paths.rkt:running-as-exe?` detects which mode is active by checking whether the `rackup-core` binary exists next to the shell wrapper. The result is cached on first call so that the mode cannot flip mid-session. Commands like `runtime install` and `runtime upgrade` become no-ops in exe mode.

The installer (`scripts/install.sh`) tries prebuilt binaries first and falls back to source. The fallback is important because prebuilt binaries only exist for a fixed set of platforms. The `binary_target()` helper maps the detected architecture to the binary artifact name (notably `arm` to `arm32`), since `detect_arch()` returns kernel names while the prebuilt tarballs use rackup's own naming convention.

The `RACKUP_PAGES_BASE_URL` environment variable overrides the base URL for all downloads (binary tarballs, source tarball, checksums). This exists for CI testing: the install.sh E2E tests serve the pages site from a local HTTP server and point `install.sh` at it.

## Shell wrapper and prompt fast-path

`bin/rackup` is a POSIX sh script that serves as the entry point. Starting Racket has measurable latency, so the wrapper handles `rackup prompt` entirely in shell code (sourced from `libexec/rackup-bootstrap.sh`) without ever launching Racket. The prompt implementation reads the default-toolchain file and parses `meta.rktd` with grep/sed to extract the version string. This keeps prompt latency under a few milliseconds.

The bootstrap helper also contains the full hidden-runtime installation logic in POSIX sh. This is necessary because when rackup is first installed from source, there is no Racket available yet to run the Racket code.

### POSIX sh vs bash

The project deliberately splits between POSIX sh and bash. `bin/rackup`, `libexec/rackup-bootstrap.sh`, and `scripts/install.sh` are POSIX sh to ensure they work everywhere — including Alpine Linux (which ships `ash`, not bash), minimal Docker images, and BusyBox environments. This matters because the installer is the entry point for new users on unknown systems. The tradeoff is some awkward POSIX workarounds: portable readlink emulation via `ls -l` string parsing, temp-file workarounds for subshell variable scoping in piped loops, and verbose IFS save/restore for PATH iteration.

The shim dispatcher (generated by `shims.rkt`) uses bash because it runs only after rackup is installed, and needs features like `[[ ]]` regex matching for toolchain ID validation, `local` function variables, and `BASH_SOURCE` for self-location. The dispatcher's security-critical regex validation (`^[A-Za-z0-9._-]+$`) has no clean POSIX sh equivalent.

## Shim dispatcher

All shimmed executables (`racket`, `raco`, `scribble`, `drracket`, etc.) are symlinks to a single bash script: `~/.rackup/libexec/rackup-shim`. The dispatcher:

1. Reads the active toolchain from `RACKUP_TOOLCHAIN` or the `default-toolchain` file. Validates the ID against the allowlist `^[A-Za-z0-9._-]+$` to prevent path traversal.
2. Sources the per-toolchain `env.sh` file if present (may set `PLTHOME`, `PLTCOLLECTS`, `PLTADDONDIR`). If `PLTADDONDIR` is still unset after sourcing, the dispatcher synthesizes a default pointing at `~/.rackup/addons/<id>`.
3. Looks up the real executable in the toolchain's `bin/` directory.
4. Falls back to addon directories (`$PLTADDONDIR/bin/`, `$PLTADDONDIR/*/bin/`) for tools installed via `raco pkg install`.
5. Runs the real binary, captures its exit status, and conditionally prints diagnostic messages for exit codes 126/127 (missing loader or executable) and 139 (QEMU/ASLR crash).

The dispatcher is generated by `shims.rkt` (the `dispatcher-script` function) and written to disk as a string literal. It is regenerated on every `reshim!` call, so changes to the dispatcher take effect on the next install/link/remove/reshim operation.

Short aliases (`r` for `racket`, `dr` for `drracket`) are opt-in, enabled via `rackup install --short-aliases` or `rackup reshim --short-aliases`. They are persisted as the `short-aliases` flag in `state/config` (a plain-text config file with one flag name per line); when set, `reshim!` creates symlinks for `r` and `dr` in the shims directory. A `case` statement at the top of the dispatcher remaps these basenames before lookup.

### 32-bit legacy toolchain support

The dispatcher detects 32-bit ELF binaries (by reading the first 5 bytes with `od`) and provides diagnostic messages when the host lacks 32-bit loader support. If `od` is not available (checked via `command -v`), the detection silently skips. It also detects qemu-i386 binfmt_misc configurations and warns about ASLR-sensitive ancient PLT Scheme releases (053, 103, 103p1) that cannot run under QEMU user-mode emulation.

## Upgrade

`rackup upgrade` upgrades channel-based toolchains (stable, pre-release, snapshot) to the latest available version. Version-pinned installs (e.g., `rackup install 8.18`, which has kind `release`) are never upgraded.

**Invocation.** `rackup upgrade` with no arguments upgrades all channel-based toolchains. `rackup upgrade stable` upgrades only stable. `--force` reinstalls even if the currently installed version matches the latest. `--no-cache` re-downloads the installer.

**Channel detection.** `upgradeable-toolchains` in `state.rkt` iterates installed toolchains and filters to those that are channel-based. The `kind` metadata field cannot distinguish stable from version-pinned installs because `release-request-hash` in `remote.rkt` hardcodes `kind` to `'release` for all release-type requests. Instead, `upgradeable-meta?` checks the `requested-spec` field (e.g., `"stable"`, `"pre-release"`, `"snapshot"`) which records the original user request. The `kind` field is still used as a fallback for pre-release and snapshot toolchains (where `kind` is `'pre-release` or `'snapshot`). An optional filter spec restricts to a single channel.

**Version comparison.** For stable and pre-release toolchains, `check-upgrade-available` in `install.rkt` resolves the latest version via `resolve-install-request` (using the same spec, variant, distribution, and architecture as the installed toolchain) and compares resolved versions using `cmp-versions`. For snapshots, the comparison uses the `snapshot-stamp` string (a date-based identifier), since snapshot versions may not change monotonically.

**Upgrade sequence.** When a newer version is available (or `--force` is set), `upgrade-toolchain!` performs the following steps in order:

1. Records whether the old toolchain is the current default.
2. Installs the new toolchain via `install-toolchain!` with options reconstructed from the old toolchain's metadata (variant, distribution, snapshot-site). This produces a new canonical ID because the version component changes (e.g., `release-9.1-cs-x86_64-linux-full` to `release-9.2-cs-x86_64-linux-full`).
3. Migrates user-scoped packages from the old toolchain to the new one.
4. If the old toolchain was the default, sets the new toolchain as default.
5. Removes the old toolchain and its addon directory.

If the install step fails, the old toolchain is untouched.

**Package migration.** `migrate-user-packages!` lists user-scoped packages from the old toolchain by running `raco pkg show --user` with `PLTADDONDIR` pointing to the old addon directory, parses the output to extract package names, then installs them in the new toolchain by running `raco pkg install --auto --skip-installed` with `PLTADDONDIR` pointing to the new addon directory. This is the same logic as `raco pkg migrate` but adapted to rackup's per-toolchain addon directory layout, where the standard `raco pkg migrate <version>` cannot locate the old packages. If package installation fails, a warning is printed but the upgrade proceeds; the new toolchain remains functional.

**Limitations of the current package migration.** Standard `raco pkg migrate <version>` preserves whether each package was explicitly installed vs. auto-installed as a dependency. The list+reinstall approach used here does not preserve this distinction. See `docs/raco-pkg-migrate-improvements.md` for a proposal to add `--from-dir` to `raco pkg migrate`, which would let rackup use the official migration tool and preserve this metadata.

## File-based locking with compile-time enforcement

Concurrent rackup processes (e.g., two shells running `rackup install` simultaneously) must not corrupt shared state. The lock mechanism uses `make-directory` as an atomic mutex: directory creation is atomic on both POSIX and Windows, so exactly one of two racing processes will succeed and the other will get `exn:fail:filesystem?`.

There are two independent locks:

- **State lock** (`~/.rackup/state/.lock`): protects the toolchain index, default-toolchain file, and shim mutations during install/remove/reshim operations. Note that `init-shell!` writes shim artifacts (the dispatcher and the core rackup symlink) without acquiring the state lock; this is intentional since init runs before any toolchains exist.
- **Runtime lock** (`~/.rackup/runtime/.lock`): protects hidden runtime installation and upgrades.

The lock implementation (`lock.rkt`) uses a `define-file-lock` macro that generates both the runtime lock and a compile-time enforcement mechanism via Racket syntax parameters. `define-file-lock` produces:

1. A syntax parameter `lock-held?` (initially `#f`).
2. A runtime function `with-name-impl` that does the `dynamic-wind`/`make-directory`/`delete-directory` dance.
3. A macro `with-name` that sets `lock-held?` to `#t` via `syntax-parameterize` and calls the runtime function.
4. A macro `define/locked-name` for defining functions that require the lock.

Functions defined with `define/state-locked` or `define/runtime-locked` expand into a pair: an `-impl` function containing the real body, and a syntax transformer that checks `(syntax-parameter-value #'lock-held?)` at every call site. If the lock is not held, `raise-syntax-error` fires at compile time. If it is held, the call compiles to a direct call to the `-impl` function with zero runtime overhead.

The `-impl` body is itself wrapped in `syntax-parameterize` so that callees also see the lock as held. This means `register-toolchain!` can call `set-default-toolchain!` without needing to re-acquire the lock. The checking is purely static: at runtime, locked functions are ordinary function calls.

The `commit-state-change!` macro in `install.rkt` wraps a body in `with-state-lock`, runs it, then calls `reshim!` to keep shims consistent with the new state.

## State storage

All rackup state lives under `RACKUP_HOME` (default `~/.rackup`). The layout is defined in `paths.rkt` as a collection of functions that compute paths from the home directory.

### Index

`~/.rackup/state/index.rktd` is the toolchain registry. It is a Racket hash with three keys:

- `installed-toolchains`: hash from toolchain ID to a metadata summary.
- `aliases`: hash from short names to toolchain IDs.
- `default-toolchain`: the default toolchain ID or `#f`.

The default toolchain is also stored separately in `~/.rackup/state/default-toolchain` as a plain text file. The separate file exists so that shell code can read the default without parsing `.rktd` format. The Racket code reads both and prefers the file.

### Config file

`~/.rackup/state/config` is a plain-text file with one flag name per line. It stores boolean configuration flags such as `short-aliases`. The `config-flag-set?`, `set-config-flag!`, and `clear-config-flag!` functions in `state.rkt` manage it. This replaced the earlier approach of separate marker files (e.g., `state/shim-aliases`) and a `config.rktd` hash; the plain-text format is simpler to inspect and edit.

### Per-toolchain metadata

Each toolchain has a `meta.rktd` file under `~/.rackup/toolchains/<id>/` recording its kind, version, variant, architecture, platform, installer URL, install timestamp, and other details. This file is written once at install time and not modified afterward.

### Toolchain install prefix (`--prefix`)

By default the on-disk directory for a toolchain is the same as its logical location, `~/.rackup/toolchains/<id>/`. With `rackup install --prefix <path>` (or `RACKUP_TOOLCHAIN_PREFIX=<path>`), the install routine places the real directory at `<path>/<id>/` and creates `~/.rackup/toolchains/<id>` as an absolute symlink pointing to it. All path-derived helpers in `paths.rkt` continue to return the symlinked location, so the rest of the codebase is oblivious to the indirection — the OS resolves through the link.

The prefix is recorded in `meta.rktd` under `'toolchain-prefix` so `rackup upgrade` reinstalls in the same prefix without the user having to repeat the flag. `delete-toolchain-dir!` (in `install.rkt`) detects the symlink form and cleans up both the link target and the link itself; `toolchain-dir-occupied?` treats a dangling link as occupied so `--force` can recover when the prefix target is gone (e.g., after `/tmp` is wiped on reboot). `rackup doctor` annotates each toolchain with its prefix or flags broken links.

### Atomic file writes

`rktd-io.rkt` provides `write-rktd-file` and `write-string-file` which write to a temporary file in the same directory and then `rename-file-or-directory` atomically into place. This prevents partial writes from corrupting state files if rackup is interrupted.

The `.rktd` reader disables `read-accept-reader`, `read-accept-lang`, and `read-accept-compiled` to prevent code execution when reading data files.

## Input validation

Several inputs from the environment are validated before use:

- **`RACKUP_HOME`** is checked for control characters in `paths.rkt:rackup-home` before any path is derived from it.
- **Toolchain IDs** (from `RACKUP_TOOLCHAIN`, the `default-toolchain` file, or install arguments) are validated against `^[A-Za-z0-9._-]+$` in `util.rkt:ensure-valid-toolchain-id!`. This is enforced in both the bash shim dispatcher and at Racket-level state read/write sites (`get-default-toolchain`, `set-default-toolchain!`, `register-toolchain!`, `unregister-toolchain!`, `resolve-active-toolchain-id`).
- **Install prefix** (`scripts/install.sh`) rejects `/`, `$HOME`, the current directory, paths with control characters, and ~20 shared system directories. The `--allow-unsafe-prefix` flag overrides this.
- **Uninstall target** rejects paths with control characters, `/`, `$HOME`, and the current directory.

## Environment variable isolation

When `bin/rackup` starts, it saves any user-set Racket environment variables (`PLTCOLLECTS`, `PLTADDONDIR`, `PLTCOMPILEDROOTS`, `PLTUSERHOME`, `RACKET_XPATCH`, `PLT_COMPILED_FILE_CHECK`) as `_RACKUP_ORIG_*` and then unsets the originals. This prevents the user's Racket environment from interfering with rackup's own runtime (whether hidden runtime or embedded exe). `PLTHOME` is not included because it is not a Racket environment variable (it is a convention from the `plt-bin` script in `racket-dev-goodies`).

When `rackup run` spawns a subprocess, `restore-saved-racket-env-vars!` in `util.rkt` first restores the user's original Racket environment variables from the `_RACKUP_ORIG_*` copies. Then `cmd-run` overlays the target toolchain's managed environment variables on top (`PLTADDONDIR`, and `PLTCOMPILEDROOTS` unless the user has set their own). This means a user-set `PLTCOLLECTS` passes through to the toolchain's Racket, allowing users to add collection directories that are available regardless of which toolchain is active.

## Per-toolchain env.sh

The shim dispatcher sources a per-toolchain `env.sh` file if present. For linked (local) toolchains, the env file is regenerated on every `reshim!` call by `regenerate-env-files!` in `shims.rkt`, because the source tree layout may change. If a local toolchain has no derivable env vars (e.g., the source root cannot be located), the env file is deleted rather than written empty. Installed (release/snapshot) toolchains do not have env files.

The env file exports all variables unconditionally (`PLTADDONDIR`, `PLTCOMPILEDROOTS`). The shim dispatcher sources env.sh per-invocation, so these are scoped to each subprocess — they do not leak into the user's shell. Neither `PLTHOME` nor `PLTCOLLECTS` is set — the Racket binary finds its own collections via compiled-in relative paths, and `PLTHOME` is not a Racket environment variable. The one exception is old PLT Scheme installations (version <= 4.x) where the parent directory is named `plt`: these set `PLTHOME` because PLT Scheme's `bin/mzscheme` shell wrapper uses it to locate the real binary under `.bin/<archsys>/`. If `PLTADDONDIR` is still unset after sourcing (or if no env file exists), the dispatcher synthesizes a fallback value (`~/.rackup/addons/<id>`).

## Per-toolchain PLTCOMPILEDROOTS

Different Racket versions (and CS vs BC) produce incompatible `.zo` bytecode. To prevent cross-version contamination in `compiled/` directories under user source trees, rackup sets `PLTCOMPILEDROOTS` per toolchain so each version+variant writes compiled output to its own subdirectory while preserving access to the toolchain's existing compiled files.

### How the value is constructed

At toolchain install or link time, `compiled-roots-value` in `state.rkt` constructs a colon-separated PLTCOMPILEDROOTS string:

1. The first entry is `compiled/<version>-<variant>` (e.g., `compiled/9.1-cs`), which is where new `.zo` files are written.
2. The remaining entries are the toolchain's existing `compiled-file-roots`, read from its `config.rktd` by `read-toolchain-compiled-file-roots`. This preserves the toolchain's native compiled-file layout:
   - **In-place installs** (rackup's default): existing roots = `(same)`, serialized as `.` → `compiled/9.1-cs:.`
   - **FHS installs** (e.g., setup-racket on CI): existing roots include an absolute reroot path → `compiled/9.1-cs:/usr/lib/racket/compiled:.`
3. `.` (equivalent to `'same`) is always included in the fallbacks so that user code's `compiled/` directories are found regardless of install layout.

The `'same` symbol in `current-compiled-file-roots` cannot be spelled directly in PLTCOMPILEDROOTS because `path-list-string->path-list` converts all entries to path objects. However, `.` is a relative path that `(build-path dir ".")` resolves to `dir` itself, which is functionally equivalent.

For linked toolchains where version or variant cannot be probed, PLTCOMPILEDROOTS is omitted. The value flows through the existing `env-vars` metadata pipeline: `write-env-file!`/`write-toolchain-env-file!` emits an unconditional export in env.sh (sourced per-invocation by the shim dispatcher), and `cmd-run` in `main.rkt` preserves a user-restored value before overlaying toolchain env vars. Racket-specific env vars are **not** exported into the user's shell by `rackup switch` — only `RACKUP_TOOLCHAIN` and `PATH` are set in the shell, to avoid leaking into non-rackup commands like `make install` in a source checkout.

### Migration for existing installs

Toolchains installed by an older rackup do not have PLTCOMPILEDROOTS in their `meta.rktd`. `regenerate-env-files!` in `shims.rkt` backfills it: on every `reshim!` call (which runs as part of `commit-state-change!` for any state-modifying command), each installed toolchain's metadata is checked, and if it lacks PLTCOMPILEDROOTS but the version+variant yield a value, the entry is added to `meta.rktd` and `env.sh` is rewritten. The backfill is idempotent — subsequent reshims see the entry and do nothing. Users only need to run any state-changing command (e.g., `rackup reshim`, `rackup install`, `rackup upgrade`) after upgrading rackup to pick up the new PLTCOMPILEDROOTS for existing toolchains.

### Cleaning up on removal

`rackup remove <toolchain> --clean-compiled` removes `compiled/<key>/` subdirectories from user package source trees before removing the toolchain. The toolchain's install directory and addon directory are fully deleted by `remove-toolchain!`, so those don't need explicit cleanup — this flag targets the directories that persist afterwards: user-scope catalog packages (which live under the addon dir) and linked packages (`raco pkg install --link <dir>`) whose source trees are anywhere on disk.

The implementation runs the toolchain's own `racket -e` with a small program that uses `pkg/lib`'s `installed-pkg-table` and `pkg-directory` to print the absolute source directory of every user-scope package, then walks each directory to find and delete `compiled/<key>/` subdirectories.

## Shell integration

`rackup init` writes a managed block into `~/.bashrc` or `~/.zshrc` delimited by marker comments (`# >>> rackup initialize >>>`). The managed block:

1. Prepends `~/.rackup/shims` to `PATH`.
2. Defines a `rackup()` shell function that intercepts `rackup switch` and `rackup shell` commands, evals the output (shell code that sets `RACKUP_TOOLCHAIN`), and delegates everything else to the real `rackup` binary.

The shell function is necessary because a child process cannot modify its parent's environment. `rackup switch stable` emits shell code like `export RACKUP_TOOLCHAIN=release-9.1-cs-x86_64-linux-full`, and the shell function evals it in the current shell. Racket-specific env vars (`PLTADDONDIR`, `PLTCOMPILEDROOTS`, `PLTHOME`) are not exported into the shell — they are set internally by the shim dispatcher via env.sh, scoped to each subprocess invocation. This prevents them from leaking into non-rackup commands.

Tab completion scripts for bash and zsh are generated by `shell.rkt` and written to `~/.rackup/shell/rackup.{bash,zsh}`. The completion functions enumerate installed toolchains by listing the `~/.rackup/toolchains/` directory.

## Installer resolution

`remote.rkt` handles all network communication for resolving and downloading toolchain installers.

**HTTP handling.** `http-open/input` follows redirects (up to 5 hops) and works with both HTTPS and HTTP URLs. Redirects are only followed to hosts in a trusted allowlist defined in `remote.rkt` (`trusted-download-hosts`); redirects to untrusted hosts are rejected. HTTP downloads are refused unless a hardcoded SHA-256 checksum is provided (`require-checksummed-http-installer!` in `util.rkt`). This matters for legacy PLT Scheme installers served from `http://download.plt-scheme.org/`.

**Installer checksum verification.** When resolving an installer, `remote.rkt` fetches the Racket release download page and parses out SHA-256 (or SHA-1 for releases before 8.2) checksums for each installer. These are included in the install request and verified after download by `verify-installer-checksum!` in `util.rkt`. The hidden runtime bootstrap path (`rackup-bootstrap.sh`) uses checksums embedded in `install.sh` at Pages build time via the `@@RACKUP_RUNTIME_CHECKSUMS@@` token.

**Version resolution:**
- `stable` queries `https://download.racket-lang.org/version.txt`.
- Numeric versions in the PLT Scheme range (up to 4.x) resolve via the hardcoded catalog in `legacy-plt-catalog.rkt`. Modern numeric versions resolve against `table.rktd` from the corresponding installer directory; if `table.rktd` is unavailable (404), the resolver falls back to scraping Apache-style directory index listings (`fetch-legacy-installer-filename`). This fallback applies mainly to Racket 5.x and early 6.x releases that predate `table.rktd`; it is rarely triggered in practice but has low maintenance cost.
- `pre-release` queries the pre-release metadata endpoint.
- `snapshot` probes both the Utah and Northwestern mirror sites, verifies installer availability on each, and picks the site with the newest `stamp.txt`. `snapshot:utah` and `snapshot:northwestern` target a specific mirror.

**Installer selection.** `select-installer-filename` (in `remote.rkt`) matches entries from `table.rktd` against the requested architecture, platform, variant (CS/BC), and distribution (full/minimal). Legacy PLT Scheme filenames use a different naming convention and are parsed/selected by `select-legacy-installer-filename` in `legacy.rkt`. When a `full` distribution is requested but unavailable, all resolution paths (release, pre-release, snapshot) automatically fall back to `minimal`.

**Distribution fallback UX.** The behavior when `full` is unavailable depends on whether the user explicitly requested it. If `--distribution full` was passed explicitly, `install-toolchain!` prompts interactively (or errors in non-interactive mode). If `full` was merely the default (no `--distribution` flag), the install proceeds with `minimal` and a warning. In both cases, after a successful minimal install, the user is shown a tip to run `raco pkg install main-distribution` to get the full distribution's packages.

**Installer execution** is platform-specific:
- Linux `.sh` installers run with `--create-dir --in-place --dest <path>`.
- macOS `.dmg` files are mounted, extracted (skipping symlinks), and unmounted.
- `.tgz` archives are extracted with the system `tar` binary. Racket has a `file/untar` library, but it is not used here because the bootstrap paths (`scripts/install.sh`, `libexec/rackup-bootstrap.sh`) must work before Racket is available. All `tar` invocations use flags compatible with both GNU tar and modern BSD tar (macOS).

Legacy interactive installers are detected by `detect-shell-installer-mode` in `legacy.rkt`, which reads the installer script body and checks for `--dest` and `--in-place` markers. Scripts with those markers use the modern flag interface; scripts that prompt for "Unix-style distribution?" or "Where do you want to install" are fed scripted stdin input.

## Version numbering

`versioning.rkt` handles version parsing and comparison. Versions are converted to a single integer for comparison using a positional scheme: `a.b.c.d` becomes `a * 10^10 + b * 10^7 + c * 10^4 + d * 10`, with an optional patch suffix (`p1`, `p2`). This supports the full range of Racket version numbers from PLT Scheme 103 through modern Racket 9.x.

`canonical-toolchain-id` generates the unique ID for installed toolchains from their kind, version, variant, architecture, platform, and distribution. For example: `release-9.1-cs-x86_64-linux-full`. Snapshot IDs additionally include the mirror site and timestamp: `snapshot-utah-2026-02-20-old1234-9.1.0.3-cs-x86_64-linux-full`.

## Legacy PLT Scheme support

Rackup can install very old PLT Scheme releases (version numbers like 103, 372, 4.2.5). The PLT Scheme version range covers versions up to 4.x; version 5.0 and later are Racket. The legacy support is factored into `legacy.rkt` and `legacy-plt-catalog.rkt`.

`legacy-plt-catalog.rkt` contains a hardcoded catalog of PLT Scheme releases. Entries with available installers include SHA-256 checksums and download URLs; some entries are reason-only records for releases that lack supported installers (e.g., "only publishes a Linux RPM"). The hardcoded catalog is necessary because these releases predate the modern `table.rktd` metadata format and are served over HTTP without checksums.

Legacy installers use different naming conventions and may require different installation procedures (e.g., interactive stdin-based installers). `detect-shell-installer-mode` inspects the installer script body to determine whether it uses the modern `--dest`/`--in-place` flag interface or expects interactive input.

## Hidden runtime bootstrap

The hidden runtime is installed by `scripts/install.sh` during source-mode installation, not by normal `bin/rackup` startup. If `bin/rackup` is run in source mode and both the hidden runtime and system `racket` are absent, it fails fast with an error. The installer calls `rackup_hidden_runtime_install_if_missing` (a shell function in `libexec/rackup-bootstrap.sh`) which:

1. Queries `download.racket-lang.org/version.txt` for the current stable version.
2. Fetches `table.rktd` and selects a minimal CS installer (falling back to BC).
3. Downloads and installs to `~/.rackup/runtime/versions/<runtime-id>/install/`.
4. Creates a `current` symlink pointing to the active version.
5. Creates a `bin` symlink pointing to the runtime's bin directory.

The entire bootstrap is in POSIX sh because there is no Racket available to run Racket code at this point. The shell wrapper then uses this hidden runtime's `racket` to run rackup's Racket source.

When rackup does have access to Racket (either hidden runtime or prebuilt exe), the hidden runtime management moves to `runtime.rkt` which handles upgrades and status checks. Both the shell-level and Racket-level runtime management use the same lock directory (`~/.rackup/runtime/.lock`) via `make-directory`/`mkdir` for mutual exclusion.

## Uninstall protocol

`rackup uninstall` must delete `~/.rackup` including the binary that is currently running. Before proceeding, the Racket code validates the target path: it rejects paths with control characters, refuses to uninstall `/`, `$HOME`, or the current directory. Unless `--dangerously-delete-without-prompting` is passed, it requires the user to type `DELETE` at an interactive prompt.

The Racket code handles confirmation, shell init block cleanup, and user-facing output. The actual `rm -rf` is done by the shell wrapper after the Racket process exits. This split is necessary because in source mode, `RACKUP_HOME` contains the running `.rkt`/`.zo` files — deleting them while Racket is still running causes load failures during process exit. No environment variables or request files are used; the wrapper simply calls `rackup_home()` to determine the path to delete.

## Cross-compilation and prebuilt binaries

Prebuilt binaries are built in CI via `.github/workflows/build-exe.yml`. Native builds run on GitHub Actions runners for x86_64-linux, aarch64-linux, x86_64-macosx, and aarch64-macosx. Cross-compiled builds for arm32-linux and i386-linux run on x86_64-linux using `raco cross`.

Cross-compilation uses a `raco cross make` step before `raco cross exe` to pre-compile `.zo` files. Without this, `raco exe` hits a Racket expander bug (fasl-read incompatible machine-type) for modules that mix `define-syntaxes` with `module*`.

Cross-compiled binaries are smoke-tested via QEMU user-mode emulation: a Docker container with the target platform is used to run `rackup version` and `rackup help`.

`scripts/build-dist.sh` drives the build: `raco exe` produces a single executable, `raco distribute` bundles it with shared libraries, and then the script assembles the final tarball layout (`rackup/bin/rackup`, `rackup/bin/rackup-core`, `rackup/lib/...`, `rackup/libexec/rackup-bootstrap.sh`).

## Testing

**Unit tests** (`test/all.rkt`) cover version parsing, installer filename matching, HTTP resolution, shell script generation, state management, shim logic, and upgrade logic (channel filtering, package show output parsing, upgrade spec derivation). The state/shims integration tests (`test/state-shims.rkt`) use `recspecs` for shell-transcript-style testing and import private functions via explicit `for-testing` submodules. The lock tests (`test/lock.rkt`) use `eval` on namespaces to verify that compile-time lock enforcement triggers correctly.

**Docker E2E tests** (`test/docker-test-fresh-install.sh`) build a Docker image and run full install/smoke tests in a disposable container. Modes include `direct` (system Racket pre-installed), `bootstrap` (install from local source checkout), and `bootstrap-curl` (install via `curl | sh` from a served pages site). The `--host-racket` flag independently controls whether the container has a system Racket (`present` or `absent`), and `--mode` controls the installation method.

**Install.sh E2E tests** verify the full `curl | sh` install path across all supported platforms. The CI job builds the pages site (with all binary tarballs), serves it via a local HTTP server, and runs `install.sh` against that server using the `RACKUP_PAGES_BASE_URL` environment variable override. Each matrix entry verifies that rackup installs in the correct mode (exe vs source), then installs a Racket toolchain and runs `racket` and `raco` through it. Platforms tested: x86_64-linux, aarch64-linux, arm32-linux (QEMU), i386-linux (QEMU), x86_64-macosx, aarch64-macosx (all exe mode), plus x86_64-linux in source-fallback mode.

**Native i386 VM tests** use QEMU system emulation to test legacy 32-bit toolchain support on a real i386 Linux kernel.

**CI pipeline** (`ci.yml`) runs unit tests on a matrix of architectures (x64, arm64, x64-macos, arm64-macos), Docker E2E tests across multiple scenarios, install.sh E2E tests across all exe platforms, and builds prebuilt binaries. All slow E2E jobs depend on `unit-tests` so they are skipped early when unit tests fail.

**Shell lint** (`shell-lint.yml`) runs `shellcheck` and `shfmt` on all shell scripts, and enforces that all GitHub Actions are pinned to commit SHAs and that write permissions are scoped to the deploy job in `pages.yml`.

**Self-upgrade verification.** `cmd-self-upgrade` downloads `install.sh` from the Pages site and verifies it against the `.sha256` sidecar file before execution.

## Pages site

The install site at `samth.github.io/rackup` is built and deployed by two workflows: `ci.yml` (build step) and `pages.yml` (deployment).

**Build.** `pages/build-pages-site.rkt` creates a source tarball via `scripts/copy-filtered-tree.sh`, computes its SHA-256, substitutes the `@@RACKUP_SRC_SHA256@@` token in `scripts/install.sh`, fetches the current stable Racket release page to extract checksums for minimal runtime installers and substitutes them into the `@@RACKUP_RUNTIME_CHECKSUMS@@` token, and generates the HTML landing page from `pages/site.rkt`. Workflow code in `ci.yml` then copies prebuilt binary tarballs into the site directory and generates SHA-256 sidecar files for each.

**Deployment.** `pages.yml` is triggered by pushes to main. It rebuilds the site, runs a `bootstrap-curl` smoke test (Docker container running `curl | sh` against the built site to verify the full install path works), and only then packages and deploys the Pages artifact.

CI verifies that every user-facing command from `rackup help` appears on the generated webpage, and that the source tarball contains no compiled artifacts.
